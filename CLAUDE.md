## Project Overview

**Project Titan Core** is a NeoForge Minecraft mod (mod ID: `projecttitancore`) targeting Minecraft 1.21.1 with NeoForge 21.1.228. The mod source lives in the `mod/` subdirectory. Java 21 is required.

## Build Commands

Run from the `mod/` directory (Git Bash):

```bash
./gradlew deployToInstance   # Build AND copy JAR to Prism instance mods folder (use this for testing changes)
./gradlew build              # Build only (compile check, output: build/libs/) — does NOT update the instance
./gradlew runClient          # Launch Minecraft client with the mod loaded
./gradlew runServer          # Launch dedicated server (headless)
./gradlew runData            # Run data generators (outputs to src/generated/resources/)
./gradlew runGameTestServer  # Run NeoForge GameTests and exit
./gradlew clean              # Clean build output
./gradlew --refresh-dependencies  # Force re-resolve all dependencies
```

> `deployToInstance` reads the mods path from `mod/local.properties` (`prism.instance.mods`). That file is gitignored and already configured for this machine.

## Architecture

### Entry Points

- **`ProjectTitanCore.java`** — Main mod class (`@Mod(MODID)`). Receives `IEventBus` via constructor injection by FML. Registers all DeferredRegisters and calls `modEventBus.addListener(...)` for lifecycle events (e.g. `RegisterCapabilitiesEvent`).
- **`ProjectTitanCoreClient.java`** — Client-only class (`@Mod(value = MODID, dist = Dist.CLIENT)`). Injects `IEventBus` in constructor and registers client-side listeners (menu screens, etc.) via `modEventBus.addListener(...)`.

### Registration Pattern

NeoForge's **Deferred Register** system is used for all game object registration. Registers are declared as static fields on the main class and passed to `modEventBus.register(...)` in the constructor. New content follows this pattern:

```java
// In ProjectTitanCore.java
public static final DeferredRegister.Blocks BLOCKS = DeferredRegister.createBlocks(MODID);
public static final DeferredRegister.Items ITEMS = DeferredRegister.createItems(MODID);

public static final DeferredBlock<Block> MY_BLOCK = BLOCKS.registerSimpleBlock("my_block", BlockBehaviour.Properties.of());
public static final DeferredItem<BlockItem> MY_BLOCK_ITEM = ITEMS.registerSimpleBlockItem(MY_BLOCK);
```

### Event Handling

- **Mod bus events** (lifecycle): `FMLCommonSetupEvent`, `BuildCreativeModeTabContentsEvent` — subscribe via `modEventBus.addListener(...)` in the constructor.
- **NeoForge bus events** (runtime): `ServerStartingEvent` — subscribe by passing `this` to `NeoForge.EVENT_BUS.register(this)` and using `@SubscribeEvent`.

### Resource & Data Files

- Language strings: `src/main/resources/assets/projecttitancore/lang/en_us.json`
- Mod metadata template: `src/main/templates/META-INF/neoforge.mods.toml` (processed at build time — edit this, not a generated copy)
- Data generator output lands in `src/generated/resources/` and is included in the mod JAR automatically via `sourceSets.main.resources`.

### Trophy Texture Generation

Item textures for trophies are generated by `scripts/gen_trophy_texture.py` (at the project root, not inside `mod/`). Run it from anywhere with `python scripts/gen_trophy_texture.py` — it writes PNGs directly into `mod/src/main/resources/assets/projecttitancore/textures/item/`.

To add a new tier texture:
1. Add an entry to `TIERS` in the script with the tier's color palette (`T`, `D`, `G`, `L`, `S`, `B` keys).
2. Add a matching 16×16 entry to `PIXEL_GRIDS` using those palette keys.
3. Run the script — it regenerates all tier PNGs.

Do not hand-edit the PNG files; always go through the script so the source of truth stays in one place.

## Diagnosing Crashes

Primary log: `C:\Users\lind3\AppData\Roaming\PrismLauncher\instances\projecttitan\minecraft\logs\latest.log`
Crash report: `...\minecraft\crash-reports\` (only written on hard crashes; `latest.log` is usually enough)

Read `latest.log` and search for `FATAL` or `ERROR` — the first one is almost always the root cause. All the subsequent `"Cowardly refusing to send event"` lines are noise caused by the earlier failure.

Common patterns:
- `has no @SubscribeEvent methods` — class has `@EventBusSubscriber` but no static `@SubscribeEvent` methods; remove the annotation until methods are added
- `does not override abstract method codec()` — all `BaseEntityBlock` subclasses need `public static final MapCodec<T> CODEC = simpleCodec(T::new)` and a `codec()` override
- `does not override abstract method quickMoveStack()` — all `AbstractContainerMenu` subclasses need `quickMoveStack`; return `ItemStack.EMPTY` as a stub when there are no inventory slots yet
- Registration errors during loading — usually a missing deferred register `.register(modEventBus)` call in the constructor

### NeoForge 21.1.x API notes

- `EventBusSubscriber.Bus.MOD` / `Bus` enum are deprecated for removal. Prefer constructor-based registration: `@Mod(dist=CLIENT)` classes can inject `IEventBus modEventBus` in their constructor and call `modEventBus.addListener(...)` directly — no `@EventBusSubscriber` needed.
- `AbstractContainerMenu` subclasses require both `stillValid(Player)` and `quickMoveStack(Player, int)` to be implemented.

## Planned Content & Integration Strategy

### Singleblock Machine — Titan Core

The Titan Core is the central progression machine. Players feed it materials, fluids, and energy to craft tier trophies that unlock the next tier of the modpack (similar to Satisfactory's milestone system). The end goal is to build a large structure in the world.

**Inventory layout:**
- 9 input slots (3×3 grid) — items
- 1 output slot — receives the crafted trophy
- 1 fluid tank (100,000 mB) with a void button
- Energy buffer (100,000 RF, accepts up to 1,000 RF/t, no export)

**Tier 1 recipe (defined target):**
- 64× dirt in each of the 9 input slots (576 dirt total)
- 10,000 mB water
- 500 RF/t constant energy input during crafting
- Output: Tier 1 Trophy (`projecttitancore:trophy_tier_1`)

**Tier trophy mechanic (planned, not yet implemented):**
- Each trophy is a custom item
- Placing it in inventory triggers a quest completion event in the Better Questing mod
- Trophies unlock the next tier of recipes

**Implementation steps:**
1. ✅ Block + BlockEntity shell, placeholder GUI
2. ✅ Full inventory (ItemStackHandler 9+1), EnergyStorage, FluidTank, capabilities, GUI with bars
3. ✅ Custom RecipeType + Tier 1 recipe JSON + crafting tick logic
4. ✅ Custom trophy item (`TrophyItem`, `trophy_tier_1`), texture via `scripts/gen_trophy_texture.py`
5. ⬜ Better Questing integration (trophy in inventory → quest completion event)
6. ⬜ Structure building in world (design TBD)

### Integration Plan (deferred — do not implement yet)

|---|---|---|

## NeoForge 21.1.x API Reference

This section documents patterns that are different from older Forge or non-obvious for this version. Verified against NeoForge 21.1.228.

### Capability System

The entire old Forge capability system (`LazyOptional`, `@CapabilityInject`, `getCapability()` override on block entities) is **gone**. The new system registers providers once at load time via `RegisterCapabilitiesEvent` on the mod bus.

```java
// In a dedicated method, called from the constructor via method reference:
modEventBus.addListener(ProjectTitanCore::registerCapabilities);

private static void registerCapabilities(RegisterCapabilitiesEvent event) {
    event.registerBlockEntity(
        Capabilities.EnergyStorage.BLOCK,
        ProjectTitanCore.TITAN_CORE_BLOCK_ENTITY.get(),
        (be, side) -> be.energyStorage   // return null to deny on a side
    );
    event.registerBlockEntity(
        Capabilities.FluidHandler.BLOCK,
        ProjectTitanCore.TITAN_CORE_BLOCK_ENTITY.get(),
        (be, side) -> be.fluidTank
    );
}
```

Querying a capability from another block:
```java
IEnergyStorage storage = level.getCapability(Capabilities.EnergyStorage.BLOCK, pos, direction);
if (storage != null) { /* use it */ }
```

Call `level.invalidateCapabilities(worldPosition)` when capability availability changes at runtime (e.g. from `FluidTank.onContentsChanged()`). This is separate from `setChanged()`.

For blocks that frequently poll a neighbour's capability every tick, use `BlockCapabilityCache` (avoids repeated map lookups):
```java
private BlockCapabilityCache<IEnergyStorage, @Nullable Direction> neighbourCache;
// initialise in onLoad(): BlockCapabilityCache.create(cap, level, neighbourPos, side, ...)
```

Built-in capability keys: `Capabilities.EnergyStorage.BLOCK`, `Capabilities.FluidHandler.BLOCK`, `Capabilities.ItemHandler.BLOCK` (and `.ENTITY`, `.ITEM` variants).

### Energy — EnergyStorage

`EnergyStorage` implements `IEnergyStorage` and `INBTSerializable`. Store as a field on the block entity.

```java
public final EnergyStorage energyStorage = new EnergyStorage(10_000, 256, 256);
// args: capacity, maxReceive, maxExtract

// In saveAdditional / loadAdditional (note: second param is new in 1.21):
@Override
protected void saveAdditional(CompoundTag tag, HolderLookup.Provider registries) {
    super.saveAdditional(tag, registries);
    tag.put("Energy", energyStorage.serializeNBT(registries));
}
@Override
protected void loadAdditional(CompoundTag tag, HolderLookup.Provider registries) {
    super.loadAdditional(tag, registries);
    energyStorage.deserializeNBT(registries, (IntTag) tag.get("Energy"));
}
```

### Fluids — FluidTank

`FluidTank` implements `IFluidHandler` but uses `writeToNBT`/`readFromNBT` (not `INBTSerializable`).

```java
public final FluidTank fluidTank = new FluidTank(16_000) {
    @Override
    protected void onContentsChanged() {
        setChanged();
        if (level != null) level.invalidateCapabilities(worldPosition);
    }
};
// Optional validator: new FluidTank(8_000, fs -> fs.is(Fluids.WATER))

// Serialization:
tag.put("Fluid", fluidTank.writeToNBT(registries, new CompoundTag()));
fluidTank.readFromNBT(registries, tag.getCompound("Fluid"));
```

### Block Entity — saveAdditional Signature

In 1.21.1 `saveAdditional` and `loadAdditional` take a second `HolderLookup.Provider registries` parameter. Always call `super` first. Keys `id`, `x`, `y`, `z`, `NeoForgeData`, `neoforge:attachments` are reserved.

### Block Entity — Ticking

```java
// In your Block class (implements EntityBlock or extends BaseEntityBlock):
@Override
public <T extends BlockEntity> @Nullable BlockEntityTicker<T> getTicker(
        Level level, BlockState state, BlockEntityType<T> type) {
    return createTickerHelper(type, ProjectTitanCore.TITAN_CORE_BLOCK_ENTITY.get(),
        TitanCoreBlockEntity::tick);
}
// tick signature: public static void tick(Level level, BlockPos pos, BlockState state, TitanCoreBlockEntity be)
```

### Block Entity — Client Sync

Two distinct mechanisms — both are needed for a complete machine:

```java
// 1. Chunk load: sends full data when chunk first loads on client
@Override
public CompoundTag getUpdateTag(HolderLookup.Provider registries) {
    CompoundTag tag = new CompoundTag();
    saveAdditional(tag, registries);
    return tag;
}

// 2. Block update: sends data on demand (e.g. after energy/fluid changes)
@Override
public @Nullable Packet<ClientGamePacketListener> getUpdatePacket() {
    return ClientboundBlockEntityDataPacket.create(this);
}
// Trigger with: level.sendBlockUpdated(worldPosition, getBlockState(), getBlockState(), Block.UPDATE_ALL);
```

### Menus — Key Patterns

- Use `FeatureFlags.VANILLA_SET` for `MenuType` — this is what NeoForge expects for standard menus.
- Use `SlotItemHandler` (from NeoForge) instead of vanilla `Slot` for block entity item slots. Vanilla `Container`-based `Slot` is deprecated by NeoForge.
- Use `ContainerData` / `SimpleContainerData` for syncing integers (energy stored, progress, etc.) server→client automatically each tick:

```java
// Server side (block entity implements ContainerData):
public int get(int index) { return index == 0 ? energyStored : 0; }
public void set(int index, int value) { if (index == 0) energyStored = value; }
public int getCount() { return 1; }

// Menu constructor:
this.addDataSlots(containerData); // synced automatically
```

- `IMenuTypeExtension.create(TitanCoreMenu::new)` if the client-side menu constructor needs a `FriendlyByteBuf` from the server. Use plain `new MenuType<>(Ctor::new, FeatureFlags.DEFAULT_FLAGS)` when no extra data is needed.

### Networking (for future use)

`SimpleChannel` is gone. New pattern:

```java
// 1. Packet as a record:
public record SyncEnergyPayload(int energy, BlockPos pos) implements CustomPacketPayload {
    public static final Type<SyncEnergyPayload> TYPE =
        new Type<>(ResourceLocation.fromNamespaceAndPath(MODID, "sync_energy"));
    public static final StreamCodec<ByteBuf, SyncEnergyPayload> STREAM_CODEC =
        StreamCodec.composite(ByteBufCodecs.VAR_INT, SyncEnergyPayload::energy,
            BlockPos.STREAM_CODEC, SyncEnergyPayload::pos, SyncEnergyPayload::new);
    @Override public Type<?> type() { return TYPE; }
}

// 2. Register via mod bus:
modEventBus.addListener(RegisterPayloadHandlersEvent.class, event ->
    event.registrar("1").playToClient(TYPE, STREAM_CODEC, ClientHandler::handle));

// 3. Send:
PacketDistributor.sendToPlayer(serverPlayer, payload);
PacketDistributor.sendToServer(payload);
```

### GUI Screens — No-Texture Pattern

This project draws the GUI programmatically (no `inventory.png` texture). Key rules:
- `renderBg` must draw **every** slot background manually — including all 36 player inventory slots (27 main + 9 hotbar). Vanilla textures that normally provide these are not present.
- Slot position formula in menu: `x = 8 + col * 18`, `y = 84 + row * 18` for main inv, `y = 142` for hotbar.
- `renderTooltip(Font, Component, int, int)` — use the single-`Component` overload. The `List<FormattedCharSequence>` overload does **not** accept `List<Component>` (type mismatch at compile time).
- `imageWidth` / `imageHeight` must be set in the screen constructor (default 176 × 166 is fine for standard layout). Slot positions in the menu are relative to the GUI origin; set them to match where you draw backgrounds.

### ItemStackHandler — Player vs Pipe Access

Do **not** override `extractItem` on the internal `ItemStackHandler` to block all extraction — this also prevents players from taking items out of input slots via the GUI. The correct approach:
- Internal handler: no restrictions (players can freely add/remove items).
- When registering `Capabilities.ItemHandler.BLOCK`, expose a **wrapper** that restricts pipe extraction from input slots (not yet implemented — defer until crafting logic lands).

### Common Gotchas

- `new ResourceLocation(ns, path)` is **private** in 1.21. Use `ResourceLocation.fromNamespaceAndPath(ns, path)` or `ResourceLocation.parse("ns:path")`.
- `BaseEntityBlock.getRenderShape()` returns `INVISIBLE` by default (for block entity renderers). Always override to return `RenderShape.MODEL` for standard cube machines.
- `@EventBusSubscriber(bus = Bus.MOD)` is deprecated. Prefer `modEventBus.addListener(...)` in the mod/client class constructor.
- `FMLJavaModLoadingContext.get()` was removed in 21.0 — mod event bus comes via constructor injection only.
